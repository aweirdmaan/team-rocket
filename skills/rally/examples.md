# Team Rocket — Examples

Concrete bad/good pairs for the failure modes in `failure-modes.md`. When prose argument doesn't land, code does. Reach for this file when reviewing or self-checking — the visual contrast usually settles the debate.

Examples are Scala-flavoured. Translate idiomatically for other languages; the shape of the smell stays. (The domain below — an invoice pipeline with per-market source types — is illustrative.)

---

## Dispatch tags

**Bad** — the trait exists only to be matched on:

```scala
sealed trait SourceType
case object CsvSource extends SourceType
case object JsonSource extends SourceType

val table: Map[Market, Map[String, SourceType]] = Map(...)

def applyOne(t: SourceType, ...): DataFrame = t match {
  case CsvSource  => Loaders.csv(...)
  case JsonSource => Loaders.json(...)
}
```

**Good** — put the function in the table:

```scala
val table: Map[Market, Map[String, (DataFrame, ...) => DataFrame]] = Map(
  Market.A -> Map(
    "stage_1" -> Loaders.csv,
    "stage_2" -> Loaders.json
  )
)
```

## Useless wrapper case classes

**Bad** — case class groups two functions for no reason:

```scala
case class StageOp(
  read: (SparkSession, String) => DataFrame,
  apply: (DataFrame, String, DataFrame) => DataFrame)

val table: Map[String, StageOp] = ...
```

**Good** — store the operation as one function with the runtime arg directly:

```scala
val table: Map[String, (SparkSession, String, DataFrame) => DataFrame] = ...
```

## Some/None pattern matching inside a transformation

**Bad** — case logic leaks into the transform:

```scala
def enrich(invoices: DataFrame, col: String, source: Option[DataFrame]): DataFrame = source match {
  case Some(lookup) => /* do the join */
  case None         => invoices.withColumn(col, lit(null).cast("string"))
}
```

**Good** — caller decides; transform assumes inputs:

```scala
def enrich(invoices: DataFrame, col: String, lookup: DataFrame): DataFrame =
  /* do the join */

// Caller:
paths.get(col).fold(invoices)(p => Loaders.enrich(invoices, col, load(p)))
```

## for-yield-if generating tests

**Bad** — N tests generated by a comprehension:

```scala
Seq(
  (1, "alpha"),
  (2, "beta"),
  (3, "gamma")
).foreach { case (stage, expected) =>
  s"label stage_$stage as $expected" in {
    stageOf(sample, stage) shouldBe expected
  }
}
```

**Good** — single test with single structural assertion:

```scala
"build a sample invoice end-to-end from CSV" in {
  sample shouldBe EnrichedInvoice(
    stage_1 = "alpha",
    stage_2 = "beta",
    stage_3 = "gamma",
    /* full expected record */)
}
```

## Helper methods wrapping one-liners

**Bad** — named for ceremony:

```scala
private def buildColumnName(stage: Int): String =
  s"metrics_totals_stage_$stage"

// later:
df.withColumn(buildColumnName(stage), col("amount"))
```

**Good** — inline the literal expression:

```scala
df.withColumn(s"metrics_totals_stage_$stage", col("amount"))
```

---

## Single-use constants

**Bad** — constant referenced once:

```scala
private val SourceKeyColumn = "KEY"
val spec = SourceSpec(path, inputField = SourceKeyColumn, ...)
```

**Good** — inline:

```scala
val spec = SourceSpec(path, inputField = "KEY", ...)
```

Keep the constant only if the value is non-obvious OR it's referenced three+ times.

## Defensive guards for impossible cases

**Bad** — production never hits this:

```scala
def initReaderIfNeeded(sourcePaths: Map[Int, String]): Unit =
  if (sourcePaths.nonEmpty) Readers.init()
```

(Production always has sources; the conditional only fires in tests.)

**Good** — trust the production contract:

```scala
Readers.init()
```

## Initial pre-fill then overwrite

**Bad** — pre-fill all six columns null, then each transform drops and re-adds:

```scala
val withPlaceholders = (1 to 6).foldLeft(invoices)((df, s) =>
  df.withColumn(s"stage_$s", lit(null).cast("string")))

// transforms then have to drop the placeholder before joining
val enriched = transforms.foldLeft(withPlaceholders) { (df, t) =>
  t.apply(df.drop(t.outputColumn))   // drop, then add
}
```

**Good** — transforms add their own columns; one post-step fills missing:

```scala
val enriched = transforms.foldLeft(invoices) { (df, t) => t.apply(df) }
val complete = allStageColumns.filterNot(enriched.columns.toSet.contains)
  .foldLeft(enriched)((df, c) => df.withColumn(c, lit(null).cast("string")))
```

## Useless intermediate variable names

**Bad** — labels for pipeline stages:

```scala
val deduped = invoices.withColumnRenamed(...)
val withDeduped = deduped.transform(...)
val enriched = withDeduped.transform(...)
val finalised = enriched.selectExpr(...)
```

(Also: "deduped" overstates it — these are renames, not the codebase's actual dedup step.)

**Good** — name what the value IS, not where it sits:

```scala
val invoices = readCsv[RawInvoice](path)
  .withColumnRenamed(...)
  .transform(...)
  .transform(...)
  .selectExpr(...)
  .as[EnrichedInvoice]
```

## Methods that wrap themselves

**Bad** — case class with read + apply, apply has internal cases:

```scala
case class StageOp(
  read: (SparkSession, String) => DataFrame,
  apply: (DataFrame, String, Option[DataFrame]) => DataFrame)

val csv = StageOp(
  read = (s, p) => Readers.read(p)(s),
  apply = {
    case (invoices, col, Some(lookup)) => /* join */
    case (invoices, col, None) => /* null */
  })
```

**Good** — caller threads read and apply directly; no wrapper, no Option:

```scala
val table: Map[String, (SparkSession, Option[String], DataFrame) => DataFrame] = Map(
  "stage_1" -> ((s, p, invoices) => p.fold(invoices)(path =>
    Loaders.csv(invoices, "stage_1", Readers.read(path)(s))))
)
```

## Currying for no reason

**Bad** — three parameter lists, no partial application ever:

```scala
def stamp(specs: Seq[SourceSpec])(idCol: String)(df: DataFrame): DataFrame = ...

stamp(allSpecs)("invoice_id")(myInvoices)
```

**Good** — one parameter list:

```scala
def stamp(specs: Seq[SourceSpec], idCol: String, df: DataFrame): DataFrame = ...

stamp(allSpecs, "invoice_id", myInvoices)
```

Curry only when the API exists to be partially applied.

## Implicit class extensions for one operation

**Bad** — `XOps` for a single use:

```scala
implicit class InvoiceOps(val invoices: DataFrame) extends AnyVal {
  def withMarketTag(market: Market): DataFrame =
    invoices.withColumn("market", lit(market.toString))
}

val tagged = myInvoices.withMarketTag(Market.A)
```

**Good** — plain function:

```scala
def withMarketTag(invoices: DataFrame, market: Market): DataFrame =
  invoices.withColumn("market", lit(market.toString))

val tagged = withMarketTag(myInvoices, Market.A)
```

Reserve implicits for operations that genuinely benefit from `.foo` chaining across many call sites.

## Misuse of vocabulary

**Bad** — naming a column-rename helper "normalize" in a codebase where `normalize` already means record deduplication:

```scala
def withNormalizedFields(invoices: DataFrame): DataFrame =
  invoices.withColumnRenamed("raw_id", "invoice_id")
      .withColumn("amount", col("amount").cast("double"))
```

(Future reader expects `normalize`-style deduplication, finds renaming.)

**Good** — inline or name what it actually does:

```scala
// Inline — four lines aren't worth abstracting:
invoices.withColumnRenamed("raw_id", "invoice_id")
    .withColumn("amount", col("amount").cast("double"))
    .withColumn("tax", col("tax").cast("double"))
    .withColumnRenamed("ccy", "currency")
```

## Generic abstractions for one concrete case

**Bad** — sealed trait for four unrelated transforms:

```scala
sealed trait StageSourceType
case object CsvSource extends StageSourceType
case class JsonSource(field: String) extends StageSourceType
case object StreamJoin extends StageSourceType
case object Passthrough extends StageSourceType
```

(They don't share a meaningful supertype. The trait exists to be a dispatch tag — see "Dispatch tags".)

**Good** — flat module of functions:

```scala
object Loaders {
  def csv(invoices, col, source): DataFrame = ...
  def json(invoices, col, src, field): DataFrame = ...
  def streamJoin(invoices, col, stream): DataFrame = ...
  def passthrough(invoices, col): DataFrame = ...
}
```

## Speculative configuration

**Bad** — CLI flag for a static fact:

```scala
opt[String]("stage-1-type")  // csv | json | ...
opt[String]("stage-1-path")
```

(The "type" of stage-1 is fixed per market in the story — Market B is always JsonSource, Market A is always CsvSource. Re-specifying via CLI is duplication.)

**Good** — encode the static fact in the codebase, take only what varies (the path) from CLI:

```scala
// CLI: only the path
opt[String]("stage-1-path")

// Static table elsewhere:
Market.B -> Map("stage_1" -> Loaders.json("lookupKey"))
Market.A -> Map("stage_1" -> Loaders.csv)
```
